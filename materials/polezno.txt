Окей, вот сводка информации из разговора, разделённая по категориям срочности и важности:

I. Срочно + Важно (Сделать в ближайшие 1.5-2 недели):

Аналитический раздел (1.5 недели):

Изучить не менее 30 релевантных источников (статьи, библиотеки, коммерческие решения и т.д.) по выбранной теме.

Оформить аналитический раздел в виде отчёта в отдельном репозитории GitLab.

Создать диаграмму IDEF0, отражающую входные и выходные данные системы.

Подготовить отдельный репозиторий со скачанными (не просто ссылками!) источниками, на которые есть ссылки в аналитическом разделе, с указанием конкретных страниц.

Залить в свой репозиторий.

Написать по крайней мере 30 осмысленных источников.

Конструкторский раздел (2 недели, параллельно с аналитическим):

Начать проработку конструкторской части: описать планируемые к реализации функции системы (без привязки к коду).

Декомпозировать IDEF0 диаграмму из аналитического раздела.

Описать используемые алгоритмы и математические модели (формулы, схемы).

Технологический раздел(2 недели):

Начать писать код.

Подготовить описание выбора языка программирования и платформы.

Добавить примеры вызова и использования кода.

Начать писать инструкции.

II. Срочно + Неважно:

Нет явных пунктов. Вся текущая работа критична по срокам.

III. Важно + Несрочно (В процессе работы и к защите):

Технологический раздел:

Подготовить инструкции для пользователей по запуску и работе с ПО.

Экспериментально-исследовательский раздел (после аналитического и технологического):

Провести эксперименты и исследования для проверки работоспособности ПО.

Если требуется, создать датасет (с доказательством его релевантности).

Сравнить своё решение с существующими аналогами (если возможно).

Организационно-правовой раздел (после технологического):

Определить требования к ПО с точки зрения законодательства и возможности коммерциализации.

Оформление (1 неделя):

Внести исправления по результатам экспериментов.

Презентация и доклад (1 неделя):

Подготовить презентацию и доклад для защиты.

Нормоконтроль
* Убедиться, что все части курсового проекта оформлены в соответствии с установленными требованиями.
* Оформление РПЗ согласно ГОСТу (использовать, возможно, "Тест ВКР").

Общие моменты:

Регулярно (раз в неделю по пятницам) встречаться с преподавателем.

Все вопросы и наработки заливать в репозиторий GitLab.

При необходимости использовать дополнительные ресурсы кафедры (тепловизор, 3D-камеры, Kinect).

Быть готовым обосновать выбор любого решения, инструмента, датасета и т.д.

IV. Неважно, но стоит помнить:

Система "Тест ВКР" для нормоконтроля работает только под русскоязычной Windows.

Возможны дополнительные консультации со специалистами по профильным темам (например, по звуку).

Защита курсового проекта предполагает выступление перед комиссией из трёх преподавателей.

РПЗ (расчётно-пояснительная записка) включает в себя все разделы и сдаётся в распечатанном и подшитом виде + электронная версия на диске(CD/DVD).

Отсутствие результата – тоже результат.

Если тема курсового проекта внезапно "закрывается" (становится секретной), её необходимо менять, даже в последний момент.

Не выкладывать дипломные работы и курсовые, из-за возможных проблем.

Gitlab админ пока не настроен, поэтому пока можно пользоваться.

Система сдачи в Gitlab. Все изменения надо заливать через Merge Request.

Важные уточнения, сделанные в ходе обсуждения:

Репозитории: Используется один репозиторий для всего курсового проекта, с тремя поддиректориями: report (отчёт), code (код), refs (источники). Ветка по умолчанию — main.

Источники: Необходимо скачивать сами источники, а не просто давать ссылки. В отчёте указывать конкретные страницы, откуда взята информация.

"Неуд" за курсовой: Приводит к смене темы и необходимости начинать всё заново.

Количество попыток защиты: Неограниченно в течение сессии и хвостовки.

Общение с преподавателем: Раз в неделю в пятницу.

Дополнительно:

Про срок: Аналитическая часть должна быть готова через полторы недели, начиная с момента разговора. Это, кажется, самый срочный этап.

Про код: Код должен быть готов к моменту окончания технологической части. Код для аналитической части не нужен.

Дедлайн: Защита курсового проекта должна быть до конца сессии.

Нормоконтроль: Нормоконтроль проходится с помощью "Тест ВКР". Если не пройти его за 2 недели, то будут приняты меры.

Расформирование: К 20-му мая.
